## 2. 핵심 원칙

```
1. 비즈니스 가치 우선 → 실장비 연동 프로젝트가 가장 높은 가치
2. 기존 코드 활용 → 새로 만들지 말고 연결
3. 점진적 고도화 → big bang 대신 단계적 개선
4. 검증 가능 → 각 단계마다 실제 동작 확인
```

## 3. 전체 개발 순서

### Phase 1: Sangfor MCP Workflow 완성 (2-3주)

**이유:** 1차 개발 완료됨, 실장비 있음, 즉시 비즈니스 가치

| 주차 | 작업 | 산출물 |
|------|------|--------|
| **1주** | 기존 MCP 38개 tools stdio 연결 | MCP 클라이언트 패키지 |
| **2주** | 웹 UI 고도화 + Excel 파싱 강화 | 업그레이드된 웹 UI |
| **3주** | 실장비 테스트 + 프로덕션 배포 | 운영 환경 배포 |

**상세:**
```
Week 1: MCP 연결
├─ sangfor-engineer-mcp의 38개 tools를 stdio로 호출
├─ 워크플로우 엔진에서 MCP tools 자동 발견
└─ 에러 핸들링 + 재시도 로직

Week 2: UI 고도화
├─ Excel 업로드 → 자동 분석 → 가이드 생성
├─ 멀티 벤더 비교 대시보드
└─ 결과물 다운로드 (DOCX/PPTX)

Week 3: 검증 + 배포
├─ 실장비(EPP/IAG/CC) 연동 테스트
├─ 프로덕션 Docker 배포
└─ 운영 매뉴얼 작성
```

**의존성:** 없음 (이미 1차 완료)

---

### Phase 2: AIOS v1 안정화 (1-2주)

**이유:** 핵심 프로젝트, 이미 운영 중, 소규모 안정화 필요

| 주차 | 작업 | 산출물 |
|------|------|--------|
| **1주** | 메일 인텔리전스 v2 안정화 | 안정화 릴리스 |
| **2주** | 문서화 + 테스트 커버리지 | 테스트 코드 |

**상세:**
```
Week 1: 안정화
├─ mail-intelligence-v2 버그 수정
├─ 성능 최적화 (쿼리 속도)
└─ 에러 핸들링 강화

Week 2: 문서화
├─ API 문서 작성
├─ 테스트 커버리지 60%+
└─ 배포 매뉴얼
```

**의존성:** 없음

---

### Phase 3: Vibe Coding OS 진행 (지속적)

**이유:** 이미 개발 진행 중, 병행 가능

| 주차 | 작업 | 산출물 |
|------|------|--------|
| **지속** | 기존 개발 속도 유지 | v5.x 기능 추가 |
| **필요 시** | 안정화 작업 | 안정화 릴리스 |

**상세:**
```

병행 전략:
├─ Sangfor MCP와 병행 개발
├─ 주 2-3일은 Vibe Coding
└─ 주 2-3일은 Sangfor MCP
```

**의존성:** 독립적

### Phase 4: F-aios-v3-core 고도화 (4-6주)

**이유:** Blueprint 완료됨, Phase 1-2 완료 후 안정화 기반에서 진행

| 주차 | 작업 | 산출물 |
|------|------|--------|
| **1-2주** | PR-10 (Langfuse) + PR-08 (벤치마크) | 관측성 + 벤치마크 |
| **3-4주** | PR-01 (LightRAG) + PR-03 (OpenSpace) | 그래프 검색 + 스킬 진화 |
| **5-6주** | PR-07 (Sandbox) + PR-02 (Mastra) | 코드 실행 + 워크플로우 |

**상세:**

```
Week 3-4: 핵심 기능
├─ LightRAG Python 서버 구축
├─ LM Studio 임베딩 연동

└─ 3가지 진화 모드 구현

Week 5-6: 워크플로우
├─ Docker 격리 샌드박스
├─ Mastra 기반 오케스트레이터
└─ 통합 테스트
```

**의존성:** Phase 1-2 완료 후 진행
